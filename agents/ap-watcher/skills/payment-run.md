# Skill: Payment Run

Prepare the weekly payment run. Select invoices due, optimize for early-pay discounts, cross-check cash impact against Treasury's 13-week projection. Surface for human authorization; never execute payments.

---

## When to invoke

Weekly — Thursday 9am ET (default; configurable for teams running other cadences). On-demand for off-cycle runs.

---

## Inputs

- All staged, validated invoices from `~/finance-data/ap/<year>-<month>/staged/`
- Vendor payment terms (from accounting MCP)
- Treasury's most recent 13-week projection (`~/finance-data/cash/<year>-<month>/projection-*.md`)
- Last week's payment run (`~/finance-data/ap/<year>-<month>/runs/`) — for "what didn't pay last week and why" continuity
- Configured thresholds and cash cushion from `config.yaml`

---

## Process

### Step 1 — Pull eligible invoices

From `~/finance-data/ap/<year>-<month>/staged/`, collect every invoice where:
- Status is "Validated" (not "Flagged")
- Due date is within the next 14 days, OR
- An early-pay discount window ends within the next 7 days

Exclude invoices marked "held" or "disputed."

### Step 2 — Compute early-pay discount opportunities

For each eligible invoice with a discount term (e.g., "2/10 Net 30"):
- Determine the discount window (e.g., 10 days from invoice date)
- Compare to today + planned pay date
- If pay date is within the window: capture the discount
- If not: skip the discount, pay at due date

Sort discount-capture opportunities by absolute dollar value (largest first). When optimizing, the runtime can defer lower-value discount captures to a later run if needed.

### Step 3 — Run the cash impact check

Total the run amount. Cross-reference Treasury's 13-week projection:
- Find the week the run impacts (typically Week 1 of the projection)
- Compute new ending cash after the run
- Check against the configured cushion (default: 60 days of expense run-rate)

If the run would push ending cash below the cushion, surface to `#finance-alerts` AND `#treasury-ops`. The human decides whether to defer some invoices to next week's run.

### Step 4 — Apply hold logic

Some invoices should never appear in a payment run:
- Disputed (per AR Follow-Up's customer dispute file, if cross-functional disputes exist)
- Above the large-invoice threshold ($10K default) without explicit prior approval
- Vendor flagged for fraud/risk review

For each held invoice, list the reason. Do not silently exclude — the human sees what was held and why.

### Step 5 — Generate the run artifact

Write the markdown file with all sections: Summary, Invoices for Payment, Cash Impact, Early-Pay Discount Capture, Held / Skipped, Items Requiring Human Review, Methodology, Audit Trail.

### Step 6 — Notify

Post summary to `#ap-ops`. Include the link to the full run file.

If cash-impact check failed: also post to `#finance-alerts` and `#treasury-ops`.

### Step 7 — Wait for human authorization

The payment run is staged but NOT executed. The human reviews the markdown file, applies any adjustments (defer X, expedite Y), and executes the payments themselves via the AP system (Bill.com, Ramp, etc.) or via the accounting system.

After human execution, the human updates the run file's status to "Executed: [date]" — this becomes input to next week's "what didn't pay last week" continuity check.

---

## Escalation thresholds

| Trigger | Channel | Action |
|---------|---------|--------|
| Run would push 13-week cash below cushion | `#finance-alerts` + `#treasury-ops` | Surface; await human decision on what to defer |
| An invoice >$10K threshold has no prior approval | `#ap-ops` | List separately under "Held / Skipped"; require human go-ahead |
| A vendor's running quarterly total exceeds an unusual baseline | `#ap-ops` | Surface as "vendor concentration check"; not a block, just visibility |
| Last week's run still has invoices marked "Executed: pending" | `#ap-ops` | Surface; investigate before this week's run |

---

## Anti-patterns to avoid

- **Don't auto-execute payments.** Ever. Even for fully-validated, well-known recurring invoices. The human pays.
- **Don't capture every early-pay discount blindly.** A discount that would breach the cash cushion isn't worth $85.
- **Don't lump held / disputed invoices in with normal invoices in the main table.** Held items go in their own section so the human can immediately see what's NOT being paid and why.
- **Don't omit the cash-impact check.** Even for small runs. The check is fast; the protection is worth it.
- **Don't run unscheduled off-cycle payments without human request.** Some teams need it occasionally (urgent vendor, missed cutoff). Make it explicit, not automatic.
