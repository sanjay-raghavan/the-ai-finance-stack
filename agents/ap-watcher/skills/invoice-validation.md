# Skill: Invoice Validation

For each incoming invoice, run a structured set of checks before staging it for payment. Catch duplicates, missing W-9s, unusual amounts, and miscoding *before* the invoice enters the payment-run queue.

---

## When to invoke

Continuously — every 30 minutes during business hours. On-demand on a specific invoice ID for re-validation.

---

## Inputs

- The new invoice (from the connected source: Gmail attachment, Bill.com record, Ramp invoice, etc.)
- Vendor master from the accounting MCP
- Trailing 12 months of invoices for this vendor
- Vendor's W-9/W-8 status (from `notion-finance` or accounting MCP)
- PO/contract data (if vendor policy requires)
- Configured thresholds from `config.yaml`

---

## Process

### Step 1 — Extract invoice metadata

From the incoming source, pull: vendor name, invoice number, date, amount, line items, PO reference (if any), sales-tax indication. If any field is unreadable, flag immediately — don't guess.

### Step 2 — Vendor match

Look up the vendor in the accounting MCP master:
- Exact name match: ✅ proceed
- Fuzzy match >85% similarity: surface as a "vendor name variant" flag — vendor may be a name change, may be a new vendor with similar name
- No match: flag as "vendor not in master — onboarding required"

### Step 3 — PO match (if required)

For vendor policies that require POs (configured per-vendor in master), validate:
- PO referenced on invoice exists in the system
- PO is open (not closed/voided)
- Invoice amount ≤ remaining PO balance

If any fails, flag.

### Step 4 — Amount-band check

Pull this vendor's invoices from the trailing 6 months. Compute mean and standard deviation. If the current invoice's amount is >2× the mean OR >3 standard deviations from the mean, flag for human review with the historical context shown.

### Step 5 — Duplicate check

The most important check. For this vendor, look for any other invoice in the trailing 30 days where:
- Amount matches within ±$5
- Date is within ±5 days of this invoice's date
- Invoice number is different (otherwise it's clearly the same invoice resubmitted)

If any match, flag as a suspected duplicate. Do NOT proceed past this check on the assumption it's coincidence — surface both invoices and ask the human.

### Step 6 — Sales-tax / 1099 treatment

For US-based vendors:
- Vendor is 1099-eligible (services, not goods)?
- W-9 on file?
- If 1099-eligible and YTD invoices to this vendor will exceed $600 with this invoice's amount included, AND W-9 missing → flag

### Step 7 — Propose coding

Look at this vendor's most-recent 12 invoices. The most common (account / dept / project) coding becomes the proposed coding. Compute confidence as the fraction of prior invoices with that coding.

If confidence < threshold (default 0.8): surface two possible codings, ask the human to pick. Remember the choice for next time (write to a vendor-coding-history file).

If the vendor was previously coded one way, but a human overrode it last time, surface the override history — don't auto-apply the original.

### Step 8 — Write the staged file + notify

If all checks pass: write the staged file, no notification (silent queue for next payment run).

If any check flagged: write the staged file with the flags surfaced, post a Slack message to `#ap-ops` with:
- One-line summary
- Link to the staged file

For duplicate suspicions: route to `#finance-alerts` immediately, not `#ap-ops`.

---

## Escalation thresholds

| Trigger | Channel | Action |
|---------|---------|--------|
| Suspected duplicate | `#finance-alerts` | Surface immediately with both invoice IDs |
| Vendor not in master | `#ap-ops` | Block staging until onboarded |
| Missing W-9 above $600 YTD threshold | `#ap-ops` | Block payment run inclusion until W-9 received |
| Amount >2× historical avg | `#ap-ops` | Surface; allow human override to include in next run |
| PO required but missing/mismatched | `#ap-ops` | Surface; block from payment run until resolved |
| Coding confidence below threshold | `#ap-ops` | Surface with two options; pick once, remember |

---

## Anti-patterns to avoid

- **Don't auto-stage on a near-duplicate.** Even "obviously different" cases need a human's eye. Duplicates are quiet errors that compound.
- **Don't trust the amount-band check on a vendor with <3 prior invoices.** Not enough history; surface as "vendor has limited history, manual review recommended."
- **Don't propose coding if confidence is below the floor.** Surface options, let the human pick. Remember and apply next time.
- **Don't silently re-code an overridden invoice.** Override history is the signal.
- **Don't skip the W-9 check.** 1099 misses at year-end are painful and avoidable.
