# Skill: Monthly Amortization

At month-end, generate the per-prepay amortization journal entry proposals for every active schedule. Hand off to Controller as part of the close.

---

## When to invoke

Day 1 of new period, 7am ET (before Controller's Pass 2 — Standard Accruals at 11am).

---

## Inputs

- Active schedules from `~/finance-data/prepayments/active-schedules.yaml`
- The just-closed period (e.g., if running on May 1, the closed period is April)
- Prior monthly-amortization outputs (for continuity / verification)

---

## Process

### Step 1 — Read the active schedules

Load `active-schedules.yaml`. Filter to schedules where:
- `status: active`
- `start_date <= last day of closed period`
- `end_date >= first day of closed period`

(I.e., the schedule was active for at least part of the closed period.)

### Step 2 — Compute the period's amortization for each schedule

Three cases per schedule:

**Case A — Full month within the schedule period:**
```
amortization = monthly_amount
```

**Case B — Schedule started mid-period:**
```
days_active_in_period = (period_end - schedule_start_date) + 1
amortization = monthly_amount × (days_active_in_period / 30)
```

(Or use actual days if the schedule specifies actual-day basis.)

**Case C — Schedule ends mid-period:**
```
days_active_in_period = (schedule_end_date - period_start) + 1
amortization = monthly_amount × (days_active_in_period / 30)
```

For all cases, round per company convention (default: nearest cent).

### Step 3 — Build the JE proposal

Generate the proposal following the canonical schema in **`stack:proposal-format`** (see [`/skills/proposal-format.md`](../../../skills/proposal-format.md)). Do not redefine the schema inline — the shared skill is the single source of truth for proposal structure, validation rules, and idempotency mechanics.

Amortization-specific specifics:

- `proposal_type: amortization`
- `auto_reverse: false` — amortization entries are not auto-reversed (the asset is consumed permanently each month)
- `originating_agent: prepay-manager`
- `originating_skill: monthly-amortization`
- `DocNumber` convention: `JE-PREPAY-<YYYY>-<MM>-<NNN>` (sequential within the period)
- `approver_routing.required_role: controller` for amounts under $10K; `cfo` for amounts at or above $10K
- `supporting_documents` should reference the schedule file and the original invoice

**Line items per schedule:**

- DR: Expense account from the schedule (with department coding)
- CR: Prepaid asset account from the schedule
- Description: Vendor + period + brief description

Never aggregate across schedules — each schedule gets its own proposal for audit clarity. Even two schedules with the same vendor and the same accounts produce separate proposals.

Run all 10 validation rules from `stack:proposal-format` before writing the file. If any rule fails, do not write the proposal — surface to `#prepay-ops` with the specific rule and the math, and halt this schedule's amortization for human resolution.

### Step 4 — Update schedule statuses

For each schedule that just had its final amortization (end_date falls in this period):
- Status: active → expired
- Update `active-schedules.yaml`

For each schedule that's still active:
- Decrement `months_remaining` (or compute from end_date)
- Update `period_recognized_through` to the current period

### Step 5 — Flag items for human review

- Schedules that just expired — recommend balance verification
- New schedules in partial first month — note the pro-rata
- Schedules with >$10K monthly amortization — surface for explicit acknowledgment
- Any schedule where the computed amount doesn't match the planned monthly_amount within ±$1 (rounding excepted) — usually indicates a data issue

### Step 6 — Write and hand off

Save the markdown to `~/finance-data/prepayments/<year>-<month>/amortization-input.md`. Controller's Pass 2 reads this file.

Post summary to `#prepay-ops`.

---

## Reconciliation check (post-close)

After Controller's close completes, run a reconciliation pass to confirm:
- The journal entries Controller posted match the proposed amortizations (no silent edits)
- The prepaid asset account balance after close = (balance before close) − (total amortization this period)
- The expense accounts received the correct DR amounts

This is the `reconciliation` cycle that runs Day 2 of close. Any drift gets flagged.

---

## Escalation thresholds

| Trigger | Channel | Action |
|---------|---------|--------|
| Active schedule with no end date in sight (>36 months remaining) | `#prepay-ops` | Surface for review — likely a data entry issue |
| Schedule's monthly amount doesn't reconcile to total/months | `#finance-alerts` | Halt — usually indicates an amendment that wasn't fully applied |
| Amortization for a single schedule >$25K | `#prepay-ops` | Material item; surface for explicit acknowledgment |
| Post-close reconciliation shows the posted JE differs from the proposal | `#finance-alerts` | Indicates Controller (or Poster) edited the JE; surface the diff |
| Schedule was supposed to amortize this period but didn't trigger | `#finance-alerts` | Schedule status may be stale; investigate |

---

## Anti-patterns to avoid

- **Don't aggregate JE rows.** Each schedule = its own row. Auditors will look at this years from now.
- **Don't round at the wrong step.** Compute precise amounts, then round only at the JE-line level. Pre-rounding compounds.
- **Don't extend an expired schedule "just one more month."** If it expired, it expired. A continuation requires a new schedule (and likely a new contract / invoice).
- **Don't skip the post-close reconciliation.** Controller (or the Poster) might edit the JEs before posting; you want to catch any drift between what you proposed and what landed in the GL.
- **Don't include schedules with `status: suspended` in the amortization.** Suspended means "pause"; resume requires an explicit reactivation.
