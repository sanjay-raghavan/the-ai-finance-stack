# Skill: Payroll Pre-Run Review

Before each payroll executes, validate the run against prior periods and the headcount plan. Surface variances. Never authorize execution; never run payroll.

---

## When to invoke

Triggered 2 hours before each scheduled payroll execution. On-demand if the CFO wants a pre-review on a manually-staged run.

---

## Inputs

- The upcoming payroll run from `gusto` MCP (or equivalent): per-employee gross pay, deductions, employer-side taxes/benefits, total net + employer cost
- Prior payroll run (the immediately preceding one of the same cadence)
- Headcount roster (active employees, with start dates, terminations, comp changes)
- Commission inputs from Revenue Ops at `~/finance-data/payroll/<year>-<month>/commission-input.md` (if applicable)
- Known-changes file at `~/finance-data/payroll/known-changes.yaml` (CFO-maintained: planned raises, bonuses, terminations)
- Configured thresholds and privacy scope from `config.yaml`

---

## Process

### Step 1 — Pull the upcoming run

From the payroll MCP, the staged-but-not-yet-executed run. Include: per-employee gross pay, deductions, employer-side costs (taxes + benefits + 401k match), total run cost.

If no run is staged (the MCP doesn't expose pending runs), pull the most recent executed run and label as "Post-run review" rather than pre-run — this is degraded mode; alert in the headline.

### Step 2 — Pull the prior period's run

For comparison. Same cadence (bi-weekly compares to bi-weekly; semi-monthly to semi-monthly).

### Step 3 — Reconcile variance to known drivers

For each driver expected to cause variance:
- **New hires** (per known-changes.yaml or new entries in roster): expected gross-pay addition
- **Terminations** (per known-changes.yaml or removals from roster): expected gross-pay reduction
- **Raises / promotions** (per known-changes.yaml): expected per-employee delta
- **Commission inclusion** (from Revenue Ops input file, if this is a commission-run period): expected addition
- **Bonus payouts** (per known-changes.yaml, if any one-time bonuses scheduled)

Sum the expected variance from these drivers. Compare to the actual MoM gross-pay change.

**Unexplained residual** = actual change − sum of explained drivers.

If residual is non-trivial (>$1K or >0.5% of gross pay), the run has unexplained variance — flag.

### Step 4 — Per-employee comp-band check

For each employee in the run, check if their comp is within the band for their role/level:
- Pull the comp band (P25–P75) for the role
- Compare the employee's base + target bonus
- If outside the band by >20%, flag as a possible data-entry error

Most flags resolve as "intentional high or low" — but the flag still surfaces. The point is to catch the occasional data-entry error before it processes.

### Step 5 — Required-doc check

For each new hire: offer letter on file?
For each termination: separation agreement on file (if company policy requires)?

If missing, flag.

### Step 6 — Commission/bonus reconciliation (if applicable)

If this run includes commission or bonus payouts:
- Cross-reference against Revenue Ops' commission input file (for commissions)
- Cross-reference against known-changes.yaml (for bonuses)
- Confirm per-employee amounts match

### Step 7 — Privacy posture check

This step is its own status item. Validate:
- All field queries were within `privacy.scope_restriction.allowed_fields`
- No queries hit `disallowed_fields`
- `#payroll-ops` channel membership hasn't changed since the prior review
- Audit log captured field-level access

If any privacy posture item failed, that's an alert-level escalation regardless of other findings.

### Step 8 — Write the review file + notify

Save the markdown. Post summary to `#payroll-ops`. Status is either:
- 🟢 **Cleared** — no flags; run can be authorized
- 🟡 **Flagged** — items require human review before authorization
- 🔴 **Privacy posture issue** — must be resolved before any payroll work continues

### Step 9 — Wait for human authorization

The human reviews the review file, resolves flags, then authorizes the actual payroll execution in the Gusto UI (or wherever). The agent does not execute payroll; never.

After human execution, the human marks the review file as "Authorized: [name, timestamp]" — this becomes input to the next review's "did everything from last time actually process?" continuity check.

## Escalation thresholds

| Trigger | Channel | Action |
|---------|---------|--------|
| Privacy posture failure | `#finance-alerts` (NOT #payroll-ops alone, since #finance-alerts captures the alarm) | Halt review; require remediation |
| Gross-pay variance >5% with no explained driver | `#payroll-ops` | Block authorization-ready signal |
| Comp-band outlier >20% (likely data error) | `#payroll-ops` | Surface; require human confirmation |
| New hire missing offer letter | `#payroll-ops` | Block authorization-ready signal |
| Termination missing separation agreement (if required by policy) | `#payroll-ops` | Block; require People Ops resolution |
| Commission/bonus amount mismatch vs. source file | `#payroll-ops` | Surface specific employee + delta |
| Pre-run review couldn't pull staged run | Alert | Degraded mode — post-run review only available |

---

## Anti-patterns to avoid

- **Don't auto-clear a flagged run.** Every flag requires human resolution.
- **Don't query fields outside the configured scope.** Privacy posture is sacred for this agent.
- **Don't share comp details in summary lines.** "Eng-A comp $185K base" is the kind of detail that stays in the file, not in the Slack summary. The Slack summary says "3 items flagged"; the file has the detail.
- **Don't normalize residuals to zero by inferring drivers.** If the variance is unexplained, surface that — don't manufacture an explanation.
- **Don't treat in-band-but-high comp as a flag.** Bands have ranges for a reason. Flag only outside the band.
