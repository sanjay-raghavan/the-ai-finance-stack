# Skill: Headcount Cost Tracking

Maintain a living roster of fully-loaded compensation cost per FTE per function. Refresh weekly. Feed FP&A's variance work and Controller's accruals.

---

## When to invoke

Weekly on Monday 9am ET. On-demand from FP&A or Controller queries ("what's the current loaded cost per engineer?").

---

## Inputs

- Current roster from payroll MCP (active employees, their base, target bonus, equity, start date, role, department)
- Benefits loading factor from `~/finance-data/payroll/benefits-load.yaml` (CFO-maintained — typically 25-35% of base depending on geography and plan generosity)
- Employer-side tax rates from the payroll MCP (or hardcoded by jurisdiction)
- 401k match policy from the same configuration
- Configured privacy scope and thresholds

---

## Process

### Step 1 — Pull current roster

From the payroll MCP, every active employee. Fields needed (within privacy scope):
- Employee identifier (anonymized as Eng-A, Eng-X, etc. for output)
- Role / level
- Department / function
- Start date
- Termination date (if any)
- Base salary
- Target bonus (if applicable)
- Equity vesting schedule (for separate SBC tracking)

NOT pulled: SSN, home address, banking, dependents, medical. Privacy scope check applies.

### Step 2 — Compute fully-loaded cost per employee

For each employee:

```
loaded_cost = base + target_bonus + (base × benefits_load_pct) + (base × employer_tax_pct) + (base × 401k_match_pct)
```

Default loading factors:
- Benefits load: 30% (configurable per geography)
- Employer payroll tax: 9.15% (FICA + state)
- 401k match: 5% (configurable per policy)

So a $200K base with $30K target bonus and standard loading:
- Base: $200K
- Target bonus: $30K
- Benefits: $60K (30% × $200K)
- Employer tax: $18.3K (9.15% × $200K)
- 401k match: $10K (5% × $200K)
- **Loaded cost: $318.3K**

### Step 3 — Roll up by function

Group employees by their department/function. For each function:
- FTE count
- Total loaded cost (annualized)
- Avg loaded cost per FTE
- Min/max range
- New hires this quarter
- Departures this quarter

### Step 4 — Compare to prior week's roster

What changed:
- New hires
- Terminations
- Comp changes (any base or bonus delta on existing employees)
- Department transfers

Each change has a $ impact on annualized cost. Sum the changes for the net week-over-week delta.

### Step 5 — Add commissions + SBC for total people cost

The base roster figure is straight payroll. Add:
- Annualized commission expense from Revenue Ops' monthly run, annualized
- Annualized SBC expense from this agent's own SBC tracking

Total people cost = loaded roster + annualized commissions + SBC.

### Step 6 — Flag strategic items

Not every roster change needs a flag, but a few patterns deserve surfacing:
- Function-share shift (e.g., Engineering went from 55% to 57% of headcount)
- Sustained comp-band creep (avg loaded cost rising faster than headcount)
- Comp changes that didn't flow through the pre-run review process

### Step 7 — Write the roster file + notify

Save to `~/finance-data/headcount/roster-current.md`. Also write `roster-<date>.md` as a snapshot — every weekly refresh is a snapshot, retained for variance reconciliation.

Post to `#payroll-ops`. Hand off to FP&A via `~/finance-data/headcount/feeds/fpa-input.md`.

## Escalation thresholds

| Trigger | Channel | Action |
|---------|---------|--------|
| Privacy scope breach | `#finance-alerts` | Halt; remediate immediately |
| Headcount net change >5 FTEs in a single week (above plan) | `#payroll-ops` | Surface; coordinate with FP&A on plan reconciliation |
| Comp change without pre-run-review trail | `#payroll-ops` | Flag specifically; data integrity concern |
| Avg loaded cost per function changed >5% WoW | `#payroll-ops` | Likely a comp-change cluster worth investigating |
| Engineering or any function share moves >2pp QoQ | `#payroll-ops` | Strategic item — surface for CFO read |

---

## Anti-patterns to avoid

- **Don't include SBC in the loaded-cost line.** It's a separate (non-cash) expense category; conflating obscures the cash-payroll picture.
- **Don't average loaded cost across functions for "company average."** The number is mostly noise; per-function detail is what's useful.
- **Don't surface individual employee compensation in the Slack summary.** File detail is OK (in the restricted channel); summary detail is not.
- **Don't update the roster with comp changes that haven't been through pre-run review.** Surface as a flagged item; let the human resolve the workflow gap.
- **Don't lose week-over-week snapshots.** Snapshots are how FP&A reconciles "what was headcount on 4/15?" months later.
