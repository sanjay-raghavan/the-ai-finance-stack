# Skill: Cash Position Snapshot

Pull live balances from every connected banking and processor MCP, categorize the cash, post a one-line heartbeat, and surface any threshold breaches as alerts.

---

## When to invoke

Every hour during business hours (8am–6pm ET on business days, configurable). On-demand from any human request ("what's our cash position right now?").

---

## Inputs

- Connected banking MCPs (`mercury`, `modern-treasury`, `plaid`) — for operating bank account balances
- Connected payment processor MCPs (`stripe`, `paypal`, `square`) — for merchant settlement float (PSP only)
- Prior hour's snapshot from `~/finance-data/cash/<year>-<month>/hourly/` — for hour-over-hour delta computation
- Yesterday's end-of-day snapshot — for day-over-day delta
- Thresholds from `config.yaml`

---

## Process

### Step 1 — Pull all banking sources in parallel

For each banking MCP, query the current balance for every account. Parallelize the API calls — the hourly snapshot should complete in <10 seconds total, not 10 seconds per MCP.

### Step 2 — Pull payment processor balances (PSP only)

If `psp_aware.enabled: true` in `config.yaml`, query the configured processor MCPs for:
- Available balance (already settled, available to transfer)
- Pending balance (in flight, not yet settled)
- Reserved balance (held by processor for chargebacks / disputes)

The total merchant float is the sum of pending + reserved across all processors. **This is never operating cash.**

### Step 3 — Categorize

Cash buckets:

- **Operating Cash** — what the company owns and can spend. Operating bank accounts (Mercury, etc.). Includes the available portion of any processor balance that's been settled and is awaiting transfer to an operating account.
- **Merchant Settlement Float** — funds held by processors pending settlement to merchants. PSP-only. Never counts toward runway or operating-cash calculations.
- **Restricted Cash** — collateral, lease deposits, security deposits, escrow. Listed separately. Not available for operations.

### Step 4 — Compute deltas

For each line:
- **Δ Hour** — current balance minus the balance from the most recent prior hourly snapshot
- **Δ Day** — current balance minus the balance from yesterday's last snapshot
- **Δ Week** — for the Friday end-of-day snapshot only, compute vs. prior Friday

If the prior snapshot is missing (e.g., it's the first run of the day), label deltas as "n/a — no prior snapshot."

### Step 5 — Evaluate thresholds

Run each pulled balance through the configured thresholds:

- **Operating cash cushion** — is operating cash above (or below) the 60-day expense run-rate floor?
- **Single-day move** — has operating cash moved >10% in the past day with no identified counterparty?
- **Burn anomaly** — is today's net cash change >2 standard deviations from the trailing-30-day average?

Each threshold that trips becomes an "Items Requiring Human Review" entry.

### Step 6 — Write the snapshot file and post

If no thresholds tripped: post the green heartbeat to `#treasury-ops`.

If any tripped: post to `#finance-alerts` with the snapshot summary plus a "Why this matters" line per breached threshold. Do NOT post the green heartbeat to `#treasury-ops` on the same hour — one channel per snapshot.

---

## Escalation thresholds

| Condition | Channel | Action |
|-----------|---------|--------|
| Operating cash below configured cushion | `#finance-alerts` | Surface with days-of-runway-remaining math |
| Single-day operating cash move >10% with no counterparty | `#finance-alerts` | Surface with last known reconciled balance |
| MCP timeout or auth failure | `#finance-alerts` within 5 min | Halt next snapshot until resolved; do not estimate |
| Any line categorization is ambiguous (account couldn't be classified as operating/float/restricted) | `#finance-alerts` | Surface with account details; request human classification |

---

## Anti-patterns to avoid

- **Never include merchant float in operating cash totals.** This is the single most damaging error in PSP treasury work. Even a "just this once" exception cascades into wrong runway figures, wrong investor reports, wrong board commentary.
- **Don't estimate cash from prior values when the MCP fails.** Halt and escalate. Wrong cash is worse than missing cash.
- **Don't smooth deltas.** If operating cash moved $500K in an hour, that's the number. Don't average it across the day.
- **Don't run the snapshot off-hours unnecessarily.** Banking MCPs are often slower outside business hours, and the heartbeat noise to `#treasury-ops` is unhelpful at 3am.
- **Don't skip the methodology section.** Audit will read it months from now to verify how a historical figure was calculated.
